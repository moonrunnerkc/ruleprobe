# RuleProbe v0.1.0 Case Study

## What We Tested

Two simulated AI coding agents were given the same task: build a REST API for managing user bookmarks using TypeScript and Express. Both agents received identical instruction files containing 10 machine-verifiable rules covering naming conventions, forbidden patterns, code structure, test requirements, and import patterns.

RuleProbe parsed the instruction file, extracted all 10 rules (zero unparseable), verified each agent's output, and produced adherence reports.

## Rules

The instruction file contained 10 rules across 5 categories:

| # | Rule | Category | Verifier |
|---|------|----------|----------|
| 1 | Use camelCase for variables and functions | naming | AST |
| 2 | Use PascalCase for types, interfaces, classes | naming | AST |
| 3 | File names must use kebab-case | naming | filesystem |
| 4 | Never use the `any` type | forbidden-pattern | AST |
| 5 | No console.log in production code | forbidden-pattern | AST |
| 6 | Named exports only, no default exports | structure | AST |
| 7 | Every public function must have JSDoc | structure | AST |
| 8 | Files must not exceed 300 lines | structure | regex |
| 9 | Every source file must have a test file | test-requirement | filesystem |
| 10 | Use relative paths, no path aliases | import-pattern | AST |

All 10 rules were extracted automatically from natural language instructions. No manual rule definitions were needed.

## Observations

1. **Same score, different failures.** Both agents achieved 70% adherence, but violated entirely different rules. Aggregate scores alone miss this. Per-rule comparison is necessary to understand where each agent struggles.

2. **Type safety is a common gap.** Agent A used `any` on two function parameters even though the instruction file explicitly said "never use any type." The types it needed (CreateBookmarkRequest, BookmarkListQuery) were already defined in its own types.ts file.

3. **Code size discipline varies.** Agent B produced a 338-line file where Agent A kept all files under 100 lines. Agent B added extra utility functions (search, import/export, tag management) beyond what the task specified, which pushed the file past the 300-line limit.

4. **Default exports sneak in.** Agent B added a `default export` for a notFoundHandler at the end of the file, violating the named-exports-only rule. This pattern is common in Express codebases, suggesting the agent followed framework conventions over project-specific rules.

5. **Test coverage is not all-or-nothing.** Agent A missed a test file for one module (middleware/validation.ts) while providing tests for everything else. Agent B covered all source files. The test-files-exist rule catches gaps that coverage percentage alone would obscure.

## Methodology

All results were generated by RuleProbe v0.1.0 running deterministic verification: AST analysis via ts-morph, file system inspection, and regex pattern matching. No LLM-based evaluation was used. The same rules file and verification engine were applied to both agents, producing repeatable results.

The raw data (JSON reports and comparison markdown) is available in the [docs/case-study-data/](case-study-data/) directory.

## Limitations

- These are simulated agent outputs, not live agent runs. The fixture code was written to represent realistic agent behavior with deliberate violations for demonstration purposes.
- Only 10 rules were tested. Real instruction files often contain 20+ rules across more categories.
- The task template (REST API endpoint) exercises a limited set of rule categories. The utility-module and react-component templates are available but were not used in this case study.
- v0.1.0 does not automate agent invocation. Users must run agents externally and point RuleProbe at the output directory.
